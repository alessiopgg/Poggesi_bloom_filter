# Bloom Filter Acceleration with AsyncIO and Multiprocessing

This project implements and compares two versions of a Bloom Filter in Python:

- Sequential baseline
- Hybrid parallel version based on AsyncIO + multiprocessing

The repository includes:

- the full source code
- a synthetic dataset generator
- a benchmark script
- a plotting script for performance analysis

The goal of the project is to evaluate the impact of a hybrid parallelization strategy on Bloom Filter performance in Python, separating I/O-bound and CPU-bound work.

For more details, see:
- `PARALLEL_FINAL_TERM.pdf`
- `elaborati_mid-term_Parallel_Programming.pdf`

## Project Overview

A Bloom Filter is a probabilistic data structure used to test set membership using a compact bit array and multiple hash functions.

This project compares:

- a **sequential implementation**, used as correctness and performance baseline
- a **hybrid implementation**, where:
  - file reading is coordinated with **AsyncIO**
  - CPU-bound hash computation and membership checks are parallelized with **ProcessPoolExecutor**

The hybrid design follows a map-reduce style approach:
- each worker processes a chunk of data independently
- local Bloom Filters are merged with bitwise OR
- query chunks are verified in parallel in read-only mode

The project investigates how performance changes with:
- dataset size
- number of worker processes
- chunk size
- dataset fragmentation across multiple files

## Dataset Generation

The datasets are **not included in the GitHub repository** and must be generated locally before running the benchmark.

This choice avoids storing very large generated benchmark files in the repository.

Datasets are synthetic and are generated by `generate_data.py`.  
Each element is a random alphanumeric string:

- lowercase letters and digits
- length uniformly distributed between 10 and 20 characters

For each dataset size, the generator creates:

- a **train dataset**, used to build the Bloom Filter
- a **test dataset**, used for membership queries

Train and test datasets are generated independently.

Generated data is stored in the following structure:

```text
bench_data/
└── size_<N>/
    └── split_<S>/
        ├── train/
        │   ├── train_00.txt
        │   ├── train_01.txt
        │   └── ...
        └── test/
            ├── test_00.txt
            ├── test_01.txt
            └── ...
```

Where:

- `size_<N>` is the total number of elements
- `split_<S>` is the number of files used to store the dataset

### Generate the datasets

```bash
python generate_data.py
```

## Implementation Notes

### Sequential Version

The sequential version is implemented in `sequential_bloom_filter.py`.

It:

- stores the filter as a `bitarray`
- uses `mmh3` to compute multiple hash positions
- supports build from one file or multiple files
- verifies membership on the test files sequentially

### Hybrid Version

The hybrid version is implemented in `hybrid_bloom_filter.py`.

It:

- uses `asyncio.to_thread(...)` to read files without blocking the event loop
- divides the dataset into chunks
- uses `ProcessPoolExecutor` for CPU-bound work
- creates one local Bloom Filter per worker during build
- merges local filters with bitwise OR
- parallelizes the query phase by distributing test chunks across processes

This design avoids direct concurrent updates to a shared bit array and reduces synchronization overhead.

---

## Measurement Methodology

The benchmark compares sequential and hybrid executions using the same dataset and the same Bloom Filter parameters.

For each configuration:

- one warm-up run is discarded
- three runs are measured
- average build time, verify time, and total time are recorded
- speedup is computed as:

```text
Speedup = T_sequential / T_hybrid
```

The reported execution time includes:

- Bloom Filter construction
- membership query phase

---

## Correctness Validation

Correctness is evaluated by comparing the hybrid implementation against the sequential baseline.

The validation checks that:

- the Bloom Filter is built successfully
- membership queries are consistent between the two versions
- no false negatives are observed

Since train and test sets are independently generated, most test queries are expected to be negative. Positive answers are interpreted as possible false positives, which are valid in Bloom Filters by design.

---

## Main Findings

According to the final report, the hybrid approach becomes clearly advantageous on large datasets.

Key observations include:

- limited gains on smaller datasets due to overhead
- strong improvements on 10M and 20M element datasets
- stable speedup close to **4×**
- best overall configuration around:
  - **8 worker processes**
  - **automatic chunk sizing**

These results show that balanced process-level parallelism can overcome the limitations imposed by Python’s GIL for CPU-bound Bloom Filter workloads.
